## 缺陷 E2E-001: 上下文注入标记泄露到语音输出

严重等级: CRITICAL
发现位置: 会话 A, Turn A08 (Is there any danger ahead?)

问题定义

Agent 将内部系统标记直接朗读给用户：
"[CONTEXT UPDATE DO NOT SPEAK] [SILENT context only, do not speak aloud]
 [VISION ANALYSIS] Scene: The image is a solid, uniform field of dark orange-red color..."

对盲人用户来说，听到 [CONTEXT UPDATE DO NOT SPEAK] 这样的元数据是极度混乱且不专业的。

代码定位

                                                                                                                                                                             1. server.py:1895 — 当 speak=False 时，vision 注入前缀 [SILENT - context only, do not speak aloud]\n
2. server.py:446-447 — context queue flush 时，全部 silent 的 item 被包裹为 [CONTEXT UPDATE - DO NOT SPEAK]\n
3. server.py:449-453 — 合并后的文本作为 role="user" 的 Content 发送到 Live API

交叉分析

这是一个依赖提示词防御的脆弱设计。整条链路是：

vision 结果 → 加上 [SILENT] 前缀 → ctx_queue.enqueue(speak=False)
    → flush → 加上 [CONTEXT UPDATE - DO NOT SPEAK] 前缀
    → send_content(role="user") → Gemini Live API

问题在于：系统完全依赖模型"理解"文本中的 [DO NOT SPEAK] 指令来避免朗读。但 Gemini Live 的 native audio 模式下，模型倾向于将所有收到的 role="user"
文本都视为需要回应的内容。当上下文窗口负载增加（A08 已经是第 8 轮），模型的 instruction following 能力下降，导致这些标记被当作可读内容处理。

根本原因: 没有在传输层面将 silent context 与 speakable content 隔离。依靠文本前缀 [DO NOT SPEAK] 来控制 LLM 行为是不可靠的。

## 缺陷 E2E-002: 工具重复调用导致响应重复

严重等级: HIGH
发现位置: 会话 A, Turn A04 (Navigate me to the nearest pharmacy please.)

问题定义

单次 "导航到最近的药店" 请求触发了 3 次 navigate_to + 1 次 nearby_search + 1 次 maps_query（共 5 个 tool 调用），产生 6 段几乎相同的语音：

"The closest pharmacy is a Walgreens 63 meters away at your 12 o'clock."  (×3)
"The closest pharmacy is a Walgreens 58 meters away, at your 10 o'clock." (×1)
"The closest pharmacy is a Walgreens 58 meters away, at your 4 o'clock." (×1)
"The closest pharmacy is a Walgreens 63 meters away at your 12 o'clock." (×1)

Agent 首次调用 navigate_to 失败 ("No walking route found")，于是模型发起 nearby_search 找到 Walgreens → 又调用 navigate_to（成功）→ maps_query 返回 CVS Pharmacy → 再次
navigate_to（CVS）。每次工具返回结果后，模型都生成一段语音描述。

代码定位

- server.py:1064+ — Function calling dispatcher: 每个 tool call 独立 dispatch，无去重或合并机制
- agents/orchestrator.py:161-165 — System prompt 中 navigate_to 描述允许模型自由决定调用次数

交叉分析

Gemini Live API 的 function calling 是流式的 — 模型可以在一个 turn 内发出多个并行或串行的 function call。当第一个 navigate_to 失败后，模型进入"重试+探索"模式：先用 nearby_search
找目标，同时用 maps_query 做更宽泛的搜索，然后对每个搜索结果分别调用 navigate_to。

根本原因: 缺少 tool call 去重/合并层。navigate_to 被调用 3 次没有任何拦截，每次结果都被注入到会话中，导致模型对每组结果都单独生成语音。对盲人用户来说，听到 6
段重复的药店方向是严重的认知噪音。

## 缺陷 E2E-003: 工具响应延迟过高

严重等级: HIGH
发现位置: 会话 A, Turn A04 (39.8s) 和 A05 (21.3s)

问题定义

┌──────┬────────────┬────────────────────────────────┐
│ Turn │    请求    │ first_agent_transcript_latency │
├──────┼────────────┼────────────────────────────────┤
│ A04  │ 导航到药店 │ 39.8 秒                        │
├──────┼────────────┼────────────────────────────────┤
│ A05  │ 纽约天气   │ 21.3 秒                        │
└──────┴────────────┴────────────────────────────────┘

用户发出请求后，等待近 40 秒才听到第一个回应。对比 A09（记忆查询）只需 3.3 秒。

代码定位

- A04: navigate_to → tools/navigation.py → Google Routes API + Google Places API (串行调用)
- A05: google_search → tools/search.py → Google Grounded Search API

交叉分析

延迟的两个来源叠加：

1. Tool execution 延迟: navigate_to 需要串行调用 Places API (nearby search) → Routes API (walking directions) → 格式化。如果第一次路线不可用还会重试。A04 的 3 次调用串行执行约
25-30s。
2. WHEN_IDLE 行为延迟: tool_behavior.py 将大部分工具标记为 WHEN_IDLE，意味着结果要等模型当前语音完成后才交付。如果模型正在说话，工具结果被 hold 住。

根本原因: 外部 API 调用无并发优化 + WHEN_IDLE 排队机制导致体感延迟 = API 延迟 × 重复调用次数 + 排队等待时间。

## 缺陷 E2E-004: 跨轮次上下文混淆

严重等级: HIGH
发现位置: 会话 B, Turn B02 (Read the menu for me please.)

问题定义

用户说 "Read the menu for me please"，Agent 首先回应：
"I'm still waiting for you to provide the destination. Where would you like to navigate to?"

这是 B01 (导航请求) 的延续状态，而非 B02 的 OCR 请求。随后 Agent 才纠正并说 "I'm not seeing any text to read at the moment."

代码定位

- server.py:449-453 — Context 以 role="user" 注入，Gemini Live 将其视为对话历史的一部分
- agents/orchestrator.py:100 — System prompt ADAPTIVE DETAIL 和上下文管理规则

交叉分析

Gemini Live API 使用持久的 WebSocket 连接，所有轮次共享同一个上下文窗口。B01 中用户说 "I need your help navigating the city"，模型进入"等待目的地"的对话状态。当 B02
的音频到达时，模型的对话状态仍然处于 B01 的导航流程中。

根本原因: Gemini Live session 是一个单一的、连续的对话流，没有 "turn boundary" 的概念。上一轮的 conversational intent（等待目的地输入）会 carry over
到下一轮，除非模型自行判断话题已切换。这在快速上下文切换时表现尤为明显。

## 缺陷 E2E-005: 工具执行后无语音输出（静默吞没）

严重等级: HIGH
发现位置: 会话 B, Turn B03 (Search for the best Italian restaurant nearby.)

问题定义

maps_query 和 google_search 均成功执行并返回了详细的意大利餐厅列表（报告中可见完整数据），但：
- agent_transcripts: 空数组
- audio_bytes_received: 0

用户完全没有收到任何语音回应。工具结果被"吞"了。

结果延迟到 B04 才被说出 — B04 用户问 "What is this building?"，Agent 先花了 27 条 transcript 说了 B03 的餐厅搜索结果，最后才回应 B04 的问题。

代码定位

- server.py:430-459 — ContextInjectionQueue.flush(): tool 结果以 role="user" content 注入，但需要等待 ModelState.IDLE 才能 flush
- server.py:472-479 — GENERATING/DRAINING 状态超时机制

交叉分析

这与 WHEN_IDLE 行为和 model state machine 紧密相关：

1. B03 的 maps_query 返回了极长的文本（11 家餐厅的详细描述，约 3000+ tokens）
2. 工具结果被注入到 context queue，标记为 WHEN_IDLE
3. 此时模型可能处于 GENERATING 状态（还在处理 B02 的 OCR 结果），queue 无法 flush
4. 在 57.3 秒的采集窗口内，工具结果始终没有被 flush 到模型
5. 到 B04 时，queue 终于 flush，之前积压的 B03 结果一次性涌出

根本原因: WHEN_IDLE + 长工具结果 + 模型忙于前一轮生成 = 结果被无限期缓冲。对于搜索这类用户明确请求的操作，用户期望几秒内得到反馈，实际等了整整一轮。

## 缺陷 E2E-006: Token 溢出导致会话崩溃

严重等级: CRITICAL
发现位置: 会话 B, Turn B05 (连接断开)

问题定义

WebSocket 收到 1011 (internal error) downstream_error，会话被 Gemini Live API 终止。根据 server 日志，错误原因是：
The input token count (133,756) exceeds the maximum number of tokens allowed (131,072)

仅 5 轮对话就达到了 128K token 上限。

代码定位

- server.py:449-453 — 所有 context 以 send_content() 追加到 Live session，没有 token budget 管理
- server.py:1682 — transcript_history = deque(maxlen=_TRANSCRIPT_HISTORY_MAX_ENTRIES) 只限制本地记录，不限制发送到 Gemini 的 token 量

交叉分析

Token 膨胀的来源（估算 B 会话 5 轮累计）：

┌──────────────────────────────────────┬──────────────────┐
│                 来源                 │   估算 tokens    │
├──────────────────────────────────────┼──────────────────┤
│ System prompt (orchestrator)         │ ~3,000           │
├──────────────────────────────────────┼──────────────────┤
│ LOD update (每轮)                    │ ~500 × 5 = 2,500 │
├──────────────────────────────────────┼──────────────────┤
│ Telemetry (每轮)                     │ ~200 × 5 = 1,000 │
├──────────────────────────────────────┼──────────────────┤
│ Vision 注入 (3 次)                   │ ~500 × 3 = 1,500 │
├──────────────────────────────────────┼──────────────────┤
│ B03 maps_query 结果                  │ ~4,000           │
├──────────────────────────────────────┼──────────────────┤
│ B03 google_search 结果               │ ~4,000           │
├──────────────────────────────────────┼──────────────────┤
│ B04 maps_query 重复出现在 transcript │ ~4,000           │
├──────────────────────────────────────┼──────────────────┤
│ 模型自身的 transcript 历史           │ ~3,000           │
├──────────────────────────────────────┼──────────────────┤
│ 工具 function call 参数 + 结果 JSON  │ ~5,000           │
├──────────────────────────────────────┼──────────────────┤
│ Gemini Live 内部 audio token 累计    │ ~100,000+        │
└──────────────────────────────────────┴──────────────────┘

关键发现：音频 token 是主要杀手。Gemini Live 的 native audio 模式中，每轮的音频输入/输出都会被编码为大量 token 保留在上下文中。5 轮 × 多段语音 ≈ 100K+ audio tokens。加上 B03
的超长搜索结果文本（~8K tokens），超出 131,072 上限。

根本原因: 没有 context window 管理策略。Gemini Live session 的 token 使用量随每轮对话线性增长（尤其是 audio
tokens），没有任何截断、摘要或滑动窗口机制。工具返回的长文本（如完整餐厅列表）加速了溢出。

## 缺陷 E2E-007: 导航工具被过度拦截

严重等级: MEDIUM
发现位置: 会话 A, Turn A01 和 A02

问题定义

模型在 A01 ("Hello, I just stepped outside") 自动调用了 get_location_info，在 A02 ("What's around me right now?") 调用了 get_location_info 和 nearby_search。两者都被
_check_navigation_intent() 拦截并返回 blocked。

{"status": "blocked", "reason": "navigation_tool_requires_explicit_user_request"}

代码定位

- server.py:1035-1050 — _check_navigation_intent(): 检查最近用户 transcript 中是否有显式导航意图关键词
- 意图检测依赖 _has_navigation_intent() 中的硬编码关键词列表

交叉分析

保护机制的设计意图是正确的 — 防止模型在用户没有要求导航时自行调用导航工具。但当用户说 "What's around me right now?" 时，这实际上是一个位置查询，nearby_search 和 get_location_info
是合理的响应。

根本原因: _has_navigation_intent() 的关键词列表过于严格。"What's around me" 不包含 "navigate"/"directions"/"how do I get to"
等导航关键词，但包含了对位置信息的隐式请求。位置查询（get_location_info, nearby_search）和导航请求（navigate_to, get_walking_directions）共享同一个拦截器，粒度不够细。

│ E2E-001 上下文泄露 │ 提示词防御不可靠      │ 所有带 SILENT vision 注入的场景    │
├────────────────────┼───────────────────────┼────────────────────────────────────┤
│ E2E-002 重复响应   │ Tool call 无去重      │ 导航、搜索等多工具组合场景         │
├────────────────────┼───────────────────────┼────────────────────────────────────┤
│ E2E-003 高延迟     │ 串行 API + WHEN_IDLE  │ 所有需要外部 API 的工具            │
├────────────────────┼───────────────────────┼────────────────────────────────────┤
│ E2E-004 跨轮混淆   │ Live API 无 turn 边界 │ 快速话题切换场景                   │
├────────────────────┼───────────────────────┼────────────────────────────────────┤
│ E2E-005 静默吞没   │ WHEN_IDLE + 长结果    │ 搜索类工具返回大量文本时           │
├────────────────────┼───────────────────────┼────────────────────────────────────┤
│ E2E-006 Token 溢出 │ 无 context 管理       │ 任何 5+ 轮的会话（尤其含工具调用） │
├────────────────────┼───────────────────────┼────────────────────────────────────┤
│ E2E-007 过度拦截   │ 意图检测粒度不足      │ 位置查询类请求                     │
└────────────────────┴───────────────────────┴────────────────────────────────────┘

核心链条: E2E-002 (重复调用) → E2E-003 (延迟叠加) → E2E-005 (结果积压) → E2E-006 (token 溢出) 形成了一条级联恶化链。重复工具调用产生更多 token → 工具结果积压更久 → 积压结果一次性
flush 时 token 暴增 → 最终击穿 context window 上限导致会话崩溃。

这意味着 E2E-006 的 token 溢出不仅是缺少 context management 的问题，更是 E2E-002/003/005 三个问题共同作用的结果。
